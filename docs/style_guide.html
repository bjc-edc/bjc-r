Pedagogic Design Principles 

  • Less text --Paul
  • Less distraction on page --Paul
  • Shorter pages --Paul
  • Project oriented --Paul
  • Pages must be doable. Please test your page by working through the page following each instruction as literally as possible. See how well we get the student to the place we intended. --Mary
  • Use any CS tool (abstraction, recursion, etc.) only where it makes sense to. Don't force anything into a project just to teach the tool; tools should be applied as they reasonably might arise. Their use has to be believable--Mary
  • Build in possible future generalizations only when it makes sense to. There are always more generalizations that could be made. When programming we only build in the ones we know we will need. We don't build in extras (just in case unless there is a good change we will need it. --Mary

Writing Conventions 

  • Words banned from the curriculum: "fun," "let's" (except in dialogs). Not quite banned, but to be avoided: "powerful."
  • A "block" is one block.
    • A block with inputs filled in is an "expression" (if the outer block is a reporter) or an "instruction" (if a command).
    • Multiple blocks strung together is a "script" or a "program."
  • We don't "put" or "build" blocks inside blocks; we "use" blocks inside blocks.
  • Lists have items, not members or elements, because of the ITEM block.
  • Reporters report (not return) values.
  • References to text within Snap! (such as the "OK" button or the "Motion" palette) should be in quotes.

HTML Conventions 

  • Use the "strong" tag for bold. Introduce new terminology using the strong tag.
  • Use the "em" tag for italics. The "em" tag is for emphasis (not vocab).
  • All text must be inside something. If it's not in a sidenote, a list, or a table, then it should be in a "p" tag. Lists themselves should be inside "p" or "div" tags. Use "div" tags for the CSS styles shown below or for groups of multiple things (like text and an image, etc.).
    • Images do not need to be wrapped in "div" tags. Images can be placed on either side using class="imageRight" or class="imageLeft" in the "img" tag. Otherwise, use class="indent" for images on their own lines (as long as they aren't massively wide).
  • Please do not use "height" or "width" HTML attributes within the img tag; this can cause deformed images in various ways (e.g., percentages vary with window-size, fixed sizing distorts if images are re-uploaded by someone else, etc.). Size the image properly before uploading.
  • Please indent code with 4 spaces (one tab) for each new level. This is so your code is readable to your colleages.
  • Use Snap<em>!</em> to write Snap!

File Naming Conventions 

Our server is cAsE SeNSiTIve so even if things look ok on your browser, they might not on bjc.edc.org. The most essential thing is that the case of the reference has to match the case of the file exactly.
  • Image files should be in all lowercase (e.g. hierarchical-3.jpg or map3x+7.png) and have no spaces (use dashes) and no weird characters like ×. Exception: abbreviations (e.g., URL-structure.jpg, TCP.png, or GPS-map.png). All images are to be stored in their unit folder in /img.
  • Snap! Project files should begin with the unit and lab numbers followed by a few Title Case words describing the project. Use dashes not spaces. (For example: U2L3-GuessNumber.xml, U3L2-DrawShape.xml, U4L3-BigNumbers.xml).) The title of the project in Snap! should be the same as the filename minus the extension, (e.g. U2L3-GuessNumber, etc.). All projects are to be stored in their unit folder in /prog.
  • Lab pages should begin with the lab page number followed by a few words that approximate the lab page title, no spaces (use dashes), all lowercase, (e.g., 1-shopping-list-app.html, 3-read-web.html, etc.).
  • TG pages should follow the same format as lab pages except starting with the lab number rather than the lab page number.
  • For all links: the references must match the filenames exactly.

KaTeX 

Please do not use the dollar-sign delimiters any more. We have changed systems.

KaTeX is our current LaTeX library, which is developed by Khan Academy.

  • You can insert inline LaTeX (such as \| f + g \| \le \|f\|+\|g\|) using a span with class="katex". For example:
    <span class="katex"> \| f + g \| \le \|f\|+\|g\|</span>
  • Centered, full-line LaTeX can be created using the syntax class="katex-block" in a span. This would render the same example on its own line like this:\| f + g \| \le \|f\|+\|g\|

Breaking CSS Blocks 

Two ways to break text where blocks are bumping into each other:

  1. Add the code <p style="clear: both; height: 0px;"></p> inside a div to keep an image from falling out of it OR to keep a side div from pushing down into the following div.
  2. Add the code style="clear: right;" to the following div if an image in white text is pushing down into a div.

How to Make a Picture of Blocks

  1. Build the script you want in snap
  2. If you want to make a big picture of a short script, choose "Zoom blocks" from the settings menu and enter "1.5"; otherwise, just leave it at "1" for single line blocks.
  3. Scroll around to locate the script
  4. Right click the script and choose "script pic..."
  5. Save the file to the unit's new image directory, e.g. /bjc-r/img/3-lists or to a shared directory, as appropriate. (If it's a script or a block with inputs that is not likely to be used again, please store the image in the unit image directory; if it's a blank block or a block with default inputs, then it should go in the "blocks" image directory.)

How to Make an Animation 

See https://www.youtube.com/watch?v=MD9wiyBlJ9Y

Or just follow these instructions (be sure to also read about animation controls below):

  1. Make a screen recording (suggested method: Quicktime Player).
  2. Open Photoshop, go to File > Import > "Video Frames to Layers," and select your movie file.
  3. Limit to every 2 or 3 frames depending on how fast your video moves (as we like to say, experiment!).
  4. If needed, show the timeline by going to Window > Timeline.
  5. If needed, change the speed by selecting all frames in your timeline (using shift), clicking the drop-down menu on one of the frames, and setting the frame delay to 0.1 seconds (or whatever looks good).
  6. Make the last few frames transparent (or add a few blank frames) so that there is a pause in the video as it loops.
  7. Go to File > "Save for Web," choose GIF, set the dimensions of the image as you want them, be sure to set it to loop forever, and save it to the image folder of your unit.
  8. Don't forget to actually link to the image... ;)

  3. NOTE: Make sure that the "class" attribute of the <img> tag is not "inline" or else the play button will land in the wrong place. If you want it to be inline, just delete the "class" attribute altogther. Using class="indent" seems to work fine.

Snap! XML file Conventions

  1. Please use "clean up" to line up all scripts neatly, then check if it's the order you most want and, if necessary, rearrange and use "clean up" again. It's just neater, and also makes sure we can see scripts even when the window is narrow.
  2. Make sure that the scripts that students are to manipulate are the ones they see when the file is loaded.
  3. Each file will have its own features, so there's no "general" style to use, but always consider which elements are essential and which merely cosmetic. In general, except when we are explicitly teaching about UI, UI is more cosmetic. (E.g., whether we initiate some action by clicking on the script, the sprite, or in some other way can be essential, but is often just a personal choice.) Please use only essential, and let students add their own personal touches.
  4. Consider when comments help and when they clutter. One or two that label things can often be a help, but you decide.

How to Link to a Project

  1. Build the script(s) you want in snap If the project contains custom blocks, be sure to not just build them, but leave them up in the scripting area. Opening a project that looks empty with a custom block buried in the palette has created confusion for folks.
  2. Choose "Export project" from the file menu
  3. Save the XML file to the unit's PROG directory, e.g. /bjc-r/prog/3-lists
  4. In the lab page HTML file where you want to link, place a picture of the script(s) wrapped with an "a href" tag linking to the XML file with class="run" in the "a href" tag. For example:

    Note that clicking the link starts a brand new project! Any work that the student has done prior to this link will NOT be available!!
Principles for when to include a script:
  • Early on, and for simple scripts (and minimal or no typing), getting them to build is useful practice.
  • For complicated scripts that we want them to test and/or modify, let them load it. Saves time and guarantees they're testing what we actually intended.
  • For lists with lots of typing, let them load initial versions, especially for getting started or experiments. Again, reduces overhead. Also, they can always tailor if they want.

How to Link to a Specfic Part of a Page Using Anchors

There are two parts of a link with an anchor (the link and the anchor):

  1. First, place the anchor on the target page:
    • <a name="picture" class="anchor">&nbsp;</a>
    • Put that &nbsp; (non-breaking space) in there, and use the whole thing (opening tag, &nbsp; and closing tag) as a space, replacing some space in the line your want to link to.
  2. Then, create a link, just like usual except add a hashtag and the anchor name to the end of the link:
    • For example: <a href="/bjc-r/docs/style_guide.html#picture">This would link to the "How to Make a Picture of Blocks"</a>, section above. Try it!

The #moveturn in the reference (a href) scrolls to that anchor (a name). (The anchor class is a workaround to scroll past the heading at the top of each lab page; it goes in the "a name" only; we hope to fix that one day.)

Dreamweaver will maintain these links over time if you move and rename files in DW. Otherwise if you move or rename a files (outside DW) you will have to fix all the links yourself.
