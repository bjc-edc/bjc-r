Building a Tic-Tac-Toe Board

On this page, you'll build an interactive display for a simple Tic-Tac-Toe game between two human players.

In later units, you'll use this as the starting point for an ongoing project working toward playing against the computer.
  1. First, load this file. It has two costumes and one brief script.
    • Read the code, and experiment by dragging the sprite around the stage to see what the script does.
    • Then clear the stage and click .
  2. Tic-Tac-Toe can be played on a board that is three rows of three squares each. You'll start by drawing a single square, then use that to draw a row of squares, and then use that to draw the entire board.
    1. If you use squares of size 100, the entire board will fit nicely on the stage.
      Build to make a single square tile. You can import and use any draw square or draw polygon block that you've created earlier, or use primitive blocks directly. Leave all control of the pen (such as pen down) outside of this block.
      Why abstraction? Why build draw a tile when you could use draw square or just the script inside it? Some projects, like building a 9x9 Sudoku board, can need more than one size of square with different thicknesses of borders. If you later decide to create a tile differently (maybe changing border or fill color), it makes sense to edit just draw a tile; changing draw square would change all squares in your project, not just board tiles.
    2. Build using your draw a tile block to draw one row of tiles, like this:

    3. Build using draw a row of tiles. It will help to pay attention to where draw a row leaves the sprite. If draw the board doesn't draw what you want, debug by testing each block by itself.
  3. Set the sprite to start drawing in the right place.
    • Experiment (or calculate) to find a good starting place that centers the board on the stage with tiles of size 100, and build set the sprite there.
    • Add that block to your draw the board block definition.
    • After draw the board finishes drawing, have it move the sprite back to its starting position at (-200, 160).
  4. To keep track of legal moves and wins, you'll need to know where each player moved. Import your between? predicate from your "U2L3-Predicates" project, and use it to write a which tile reporter. (It'll require more than one if block.) Test it by having the sprite say which square it's in for a second when dropped.
    Stuck on debugging which tile? Click here for one common bug.
  5. Once you have which tile debugged, instead of saying the tile number, move the sprite to the center of the tile before stamping:
    1. Make a global variable named centers.
    2. In the same script that draws the board to start a game, set centers to a list of nine points (using the abstract data type from Lab 2), each of which is the center of the corresponding tile.
    3. In the when I am dropped script, if the sprite was dropped in a tile, move to its center before stamping it. (If it wasn't dropped inside a tile, don't stamp it and don't change costumes.)
  6. Check for illegal moves. You're already checking for dropping the sprite outside the board, but it's also illegal to drop the sprite inside a tile that's already occupied. To check for this, you'll have to keep track of the moves:
    1. Make a global variable board.
    2. In the initialization script, set it to a list of nine items, each of which is the word Empty.
    3. When testing that the sprite is dropped in a tile, also test whether the corresponding item of board is Empty. If not, don't accept the move.
    4. If the move is legal, set the corresponding item of board to either X or O. You can check the to decide which.
