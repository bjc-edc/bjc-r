Debugging Recap

On this page, you will review what you've learned about debugging in Snap! and learn some good programming habits.

The first computer "bug" was an actual moth that got stuck inside a computer in 1947.

Debugging is the art of identifying errors in computer programs and finding ways of resolving them.

Dos and Don'ts

  • Do reproduce the problem several times so you are sure you know how to recreate it. This will help you know when you've fixed it.
  • Do isolate the piece of code producing the error. Test pieces of your code separately so you know what's working and what isn't.
  • Do identify the specific cause of the error within that code. Why is that code doing the wrong thing?

  • Don't delete your project and start over. This is a waste of your time and all of your code that is working. You might end up with the same bug again, and you might end up with new bugs in parts of the program that you had working before.
  • Don't add code to work around the bug. Actually fix the bug. Think: "debug by subtraction not by addition."
  • Don't be hard on yourself. Everyone gets bugs in their programs; it's part of programming.

Tips

  • Always test your code with different inputs. What if the input is really big, zero, negative, or a decimal? What if the user types something you aren't expecting? What if the user clicks somewhere else?

  • Keep a list of bugs you encounter and your solutions. The same bugs may come up again in a slightly different way, and your record will make fixing a similar bug next time less frustrating. Include any test cases that helped you identify the bugs (for example, "try inputting a word instead of a number").
  • Love your bugs! They pose an interesting puzzle, and help you become a better programmer.

Good Habits to Minimize Bugs

  • Document your program using comments.
    You learned about Snap!comments in Unit 1 Lab 3 Page 3: Blocks with Inputs.
  • Keep your code tidy. Delete unused blocks, and use the Snap!'s "clean up" feature to organize your code.
    You learned about code organization in Unit 1 Lab 3 Page 4: Modify Your Pinwheel
  • Use readable names for blocks and variables.
    In text-based programming languages (Python, Java, etc.) you may not be able to use spaces in variable or procedure names, and because programmers have to write the name out each time, it's common to abbreviate names. But this makes the names harder to read and understand what they mean. For example in Snap!, you might use a variable name like Alonzo's Current Score, but in a text-based language, you might end up with aloCurScr. Take advantage of Snap!'s flexibility with naming, and make your blocks and variables with sensible names that you will remember later and other people can understand easily.
  • Use abstraction to hide the complexity of code. Examples of abstraction include: using loops to repeat, using custom blocks, using an abstract data type.
