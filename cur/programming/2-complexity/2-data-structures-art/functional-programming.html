Functional Programming

In the language used on the exam, the only way to generate list data is with commands, actions that change the contents of a list: Add. Insert. Replace. Delete. They take this approach because they think it's the easiest programming style for you to understand, and also to include Scratch, in which these commands are the only mechanism provided for working with lists. But there's a better way, called functional programming, in which you never modify a data structure, but instead make a new one to hold the new values.

I've yet to be convinced that it's always better. Certainly, in some cases it is, but sometimes, it's a big ugly confusing mess. At one time, functional programming was viewed by "real" programmers as something of interest only to computer science professors. But today it's a favored way to program because of parallel-processing hardware, either with racks full of computers or with multicore processor chips that include four or eight processors inside a single chip. The reason is that with multiple processes running in parallel, it's possible for two of them to be trying to modify the same variable at the same time, with bad results. But if the program is written functionally, nothing is ever modified, so that can't happen. There are functional languages in which there are no equivalents to set or replace.
