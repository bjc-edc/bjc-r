    This video is very nice and suggests a demo that teachers might want to do in class. But, the more I think about it, the more I think it complcates the basic idea. You don't need two separate keys to *send* a secure message.

    The message we'd like to get across is that the security of the system lies in the difficulty of inverting a function. The demos show how two people can share a secret, but it doesn't get at this essential feature, I think.

    The big picture is that Alice has a 1-1 function f that's next to impossible to invert, and she posts the function for everyone to see. She (and only she, for the foreseeable future) knows f^{-1}. Bob wants to send Alice a message M, so he posts, for all the world to see, f(M). Alice gets the encrypted message and decodes it by calculating f^{-1}\left(f(M)\right).

    The original way one constructed the function f was to use something like x\mapsto x^e \mod N Where N is a product of 2 huge primes p and q and e is relatively prime to (p-1)(q-1). The public key is then the pair (N,e) and the secret key is the number d so that ed \equiv 1 (p-1)(q-1).

    The use of both private keys comes in when you want to have an electronic signature to add even more security. Then Bob first encrypts his message to Alice with his private key, then Alice's public key. Alice gets the message and decrypts it by applying her private key and then Bob's public key. All this is a nice application of function composition and how to apply functions in certain sequences.
