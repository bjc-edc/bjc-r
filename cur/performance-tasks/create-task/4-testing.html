Testing Your Project

On this page, you will reflect on your specifications to design a testing plan, and then you'll run the tests and fix any errors.

CRD-2.J.1, CRD-2.J.2, CRD-2.J.3

After you develop a prototype, you'll need to test it and revise it based on the results. But testing is more than "just playing with the program as a user." Good testing is systematic, and it requires referring back to the program requirements to identify a variety of inputs to test different possibilities for user behavior (including extreme cases, such as very long words, very small numbers, very fast clicks, etc.) and the expected outcomes for these inputs.

    CRD-2.J
  1. Looking back at your specifications, identify: the types of inputs your program takes, a variety of different and extreme cases for these inputs, and the expected outcomes for each case. Record these testing inputs and their expected outcomes.

What you'll need to do to fix errors in your program depends on what type of error occurs:

CRD-2.I.1, CRD-2.I.2, CRD-2.I.3, CRD-2.I.4
  • A logic error causes the program to produce the wrong output. (This is the most common error in Snap!)
  • A syntax error means the rules of the programming language were not followed. (These are more common in text-based languages because you can easily mistype something or forget a needed semi-colon or quotation mark. By comparison, Snap! doesn't let you drop a block where it doesn't belong.)
  • A run-time error an error that happens while your program is running. What counts as a run-time error depends on the program you are using; in Snap! these are the errors that stop your program with a red halo surrounding the block that caused the error.
  • An overflow error happens when a number is too extreme for the computer to handle (too big or too small).

CRD-2.I.5

There are many ways to find and correct errors. Here are several common onces:

  • Test Cases—identifying extreme input cases to test and their expected outcomes (This is the primary method you'll be using.)
  • Hand tracing—reading the code by hand and testing various cases as though you were the computer
  • Visualizations—creating a visual way to see what your program is doing (such as a variable watcher, or a chart of inputs, expected outcomes, and actual outcomes)
  • Debuggers—some programming languages include a debugging feature that will help you locate and understand errors (such as Visible Stepping in Snap!)
  • Adding Output(s)—you can add say blocks to your code or add temporary variable watchers to help you understand what values are held by your program at different times (this is another great strategy in Snap!)
