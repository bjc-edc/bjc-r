Showing Your Code for the AP Create Task

Snap! has two convenient ways to share some or all of your code.

Creating Script Pics

You can download a PNG image file of any block or script by right-clicking it (or control-clicking it on a mac) and selecting "script pic..." If a new window opens with the picture, use your browser's Save command (in the File menu) to save it. If not, look in your browser's Downloads folder for the image file.

A block definition in the Block Editor A toplevel script If you shift-right-click (or shift-control-click) on a reporter block, you will get a script pic of the reporter with its report bubble showing.

Exporting All the Images at Once

You can download an HTML file of all of the blocks in your whole project as well as a picture of the stage by selecting "Export summary..." from the Snap! File menu (). The HTML file also land in your browser's Downloads folder. Open it (in a browser), right-click (or control-click on a mac) to copy the images you need, and paste them into the document where you are describing your code.

Abstraction

The last step shown above, using repeat to clarify the structure is also a simple example of abstraction. It hides a lot of repetitive detail and shows the structure better. This type of abstraction is not what the AP Create Task is looking for though.

Loops are an example of abstraction, but the expectation for the Create Task is that you create and submit your own example of abstraction rather than just pointing out existing abstractions like loop blocks.

When you create a block and give it a name that describes its purpose, you are creating an abstraction. You are hiding the details and showing only the purpose. When you use that pinwheel block, it shows the structure of your program more clearly. Someone reading your program doesn't need to figure out what the details inside pinwheel do; they can tell from the name that it makes a pinwheel with a given number of branches.

Note that while packing your entire program into a single block might the page cleaner, it is not a good example of abstraction. The best examples of abstraction to highlight for the AP are pieces of code that you use multiple times in your program and are clearer because they are packed into a well-named block. You can often use what's in that block as an example of the algorithm for the purpose that the block serves.

Gotchas

Definitely read the grading scale from the College Board. (This link is to the 2019 grading instructions, but it's close enough.) You will, we hope, be amused that you don't get any points for a working program. Only the snippet that you mark as your algorithm for submission purposes has to work. But there are other grading issues that might not be obvious. Here's one:

They want you to show that you went through a process of repeatedly making changes in your code to fix bugs or add features. The instructions for students say, "Describe the incremental and iterative development process of your program, focusing on two distinct points in that process." What they mean is that you have to describe two times when you fixed or improved something, not two code pictures showing the before and after states of a single improvement. Many students have lost these two points because of this misunderstanding.

Why don't they grade more sensibly?

In a world with very competitive college admission, full of parents with lawyers, it's very important to the College Board that scores be reliable, which means that you get the same score regardless of who grades your submission. The week before the actual grading, they have training sessions in which graders read sample submissions from previous years, and must give the correct grades for each. That kind of reliability is achievable for tiny yes-or-no questions such as "Are there two examples of refining the program?" but not achievable for broader questions such as "How well does the program work?" or "Does the program do something sensible?"
