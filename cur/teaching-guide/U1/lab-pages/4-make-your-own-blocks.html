Programming Lab 4: Building and Debugging Your Own Blocks

Students learn to create custom blocks and blocks with inputs and to use custom blocks within custom blocks. Using blocks within blocks presages recursion (calling a procedure from within itself), which will be addressed in Unit 5: Recursion and Unit 6: Recursive Reporters. In this lab, students continue to develop comfort with block programming and the snap interface as they create their own blocks and identify categories and input variables for their new blocks.

Students also build their skill with debugging, and they consider how a block that works well on its own might cause problems when used within another block.

Pacing

The 5 lab pages could be split across two or three days (55-80 minutes). Expected times to complete Lab 4 follow:

  • 10-15 minutes on Designing Custom Blocks
  • 5-10 minutes on Adding an Input
  • 15-20 minutes on Debugging Custom Blocks
  • 15-20 minutes on Using Blocks within Blocks

Tips:

  • A very common error, even among experienced snap programmers, is to forget to specify the palette/color when making a new block. The block then ends up in the gray "Other" category, which is especially hard to find because it shares a palette with Variables. To change a block's palette, open it in the block editor and click on the orange hat block at the top of its script.
  • It's also possible to change the shape of a custom block in the block editor, but only if there are no invocations of the block anywhere in a script, whether in the scripting area (including the block by itself) or in a custom block. You have to delete all such invocations (by dragging them onto the palette area) before clicking on the hat block.
  • The draw polygon problem in the Take It Further on the Adding an Input page is really important; you might want to assign it to everyone. It introduces the idea of generalizing patterns: If you have a family of custom blocks that are mostly the same script but with a little difference in each one, you can make a single block that captures the entire family by adding an input for the part that's different. We'll use this idea many times in the course, but it's especially important in Unit 5 when the result of the generalization is a recursive procedure.
  • Another point about adding inputs is that the custom blocks your students create on this page will have rectangular input slots, not oval as in the primitive blocks, for numeric inputs. If a student questions this, say that this won't affect the usability of their blocks, but that we'll get to distinguishing input types in Unit 2, Lab 3.

  • About bugs: Beginning programmers often get very upset and discouraged when they see an error message. But experienced programmers are happy about error messages, because they're much easier to fix than the kind of bug that runs to completion without an error message, but does the wrong thing. The error message helps pinpoint where in the code the bug lies. You can say that to students who seem frustrated by an error message.
