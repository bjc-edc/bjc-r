Lab 2: Gossip and Greet

In Labs 2 and 3, students learn a new idea: not only can they assemble existing blocks (code) to create sequences (scripts), they can also encapsulate an entire script into a single new block of their own design, which they can then use as if that block had always existed. This process—one aspect of abstraction—is basic to computer science. Students will see how a multi-part script can be "hidden" inside a block (so students don't need to see all the details of the block at once) and given a name (so they can easily know what it does).

In this Lab 2, students program two sprites to gossip with each other, picking randomly from words and ideas that students specify. The context is whimsical—students generally enjoy teaching a computer to gossip and find the results funny—but the ideas they are learning are deep. Getting a computer to generate (and understand) natural language has been the serious work of computer scientists for decades; Siri is one of the results. They first experiment with blocks that have been built for them and see what they do. Then, they learn to edit and modify these blocks. Finally, they create a new block of their own using the exact same structure they've seen in the blocks they've edited, but with new content. In doing so, they will learn how to create lists of data, select data from those lists, and combine data in new ways. In Lab 3, they will extend this knowledge to another aspect of abstraction—learning how to give blocks inputs.

Pacing

The 4 lab pages could be split across 2-3 days (55-110 minutes). Expected times to complete follow:
  • 10-20 minutes (<1 class period) on Making Programs Talk
  • 15-30 minutes (<1 class period) on Customizing and Debugging
  • 20-40 minutes (<1 class period) on Making Your Own Block
  • 10-20 minutes (<1 class period) on Learning Player Names

Lab Pages

  • Page 1: Making Programs Talk.
    • Learning Goals:
      • Learn to try out code to see what it does.
      • Analyze code to see how it works.
    • Tips:
      • The XML file that students load (U1L2-GreetPlayer.xml) already has a player list variable that has been initialized as an empty list. In future units, students will need to define and initialize variables themselves, but as this unit is not focused on the use of variables, it has already been done in this project.
      • The starter project shows Snap! code with comments attached. Some students may want to comment some of their code, for their own information, but please don't require students to do so. This isn't a software engineering course preparing students for the workforce. Keep the focus in this first course on the beauty and joy.
  • Page 2: Customizing and Debugging.
    • Learning Goals:
      • Personalize the who, does what, and who2 blocks by inserting the phrases into the list blocks inside.
      • Become familiar with the concept of debugging, and debug any issues that arise in the project.
    • Tips:
      • Set clear expectations for the appropriateness of the student additions to the gossip project blocks.
      • It is probably too early to talk about abstraction and name it. Students will need to see more examples of it before it needs a piece of technical vocabulary. But two of its attributes may be worth mentioning even now: one, putting inside a block named makes code easier to read and understand by "hiding the complexity," and two, using a simple name that describes the purpose of the code. Once we understand what who does and are assured that it works properly, we no longer need or want to see the guts of how it works.
  • Page 3: Making Your Own Block.
    • Learning Goals:
      • Create a new reporter block (oval shape) called gossip response with the same inner structure as the three reporters included in the starter project.
      • Edit the gossip2 block to include the new block.
  • Page 4: Learning Player Names.
    • Learning Goal: Create a new command block (puzzle-shaped) called greet player that greets the player appropriately when clicked.
    • Tip: In the starter project included in this lab, the player list variable has already been created and initialized as an empty list. In Unit 2, students will learn how to create and initialize variables, but in this early lab, we've done this for students so that they can focus on sequencing the blocks provided. You can explain this if someone asks, but learning about variables is not a goal of this lab and does not need teaching at this time.

Essential Understandings:

Students edit and create custom blocks that hide the details of the underlying code (EU 2.2, EU 5.3). This lab builds toward these EUs by introducing abstraction in the context of conversation: the program constructs a sentence by appending a noun phrase, a verb phrase, and a second noun phrase, and each phrase is itself another procedure that selects its output randomly from a list prescribed by the student.
  • EU 2.2 Multiple levels of abstraction are used to write programs or create other computational artifacts.
  • EU 5.3 Programming is facilitated by appropriate abstractions.

Learning Objectives:

Following the model blocks provided in the starter project for this early lab, students create custom noun block (LO 2.2.1) that masks the complexity (LO 5.3.1) of the program.
  • LO 2.2.1 Develop an abstraction when writing a program or creating other computational artifacts. [P2]
  • LO 5.3.1 Use abstraction to manage complexity in programs. [P3]

Essential Knowledge:

  • EK 2.2.1A The process of developing an abstraction involves removing detail and generalizing functionality.
  • EK 2.2.1B An abstraction extracts common features from specific examples in order to generalize concepts.
  • EK 5.3.1A Procedures are reusable programming abstractions.
  • EK 5.3.1C Procedures reduce the complexity of writing and maintaining programs.
  • EK 5.3.1D Procedures have names and may have parameters and return values.
  • EK 5.3.1I Strings and string operations, including concatenation and some form of substring, are common in many programs.
