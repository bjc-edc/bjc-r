Optional Projects

Pacing

Expected times to complete the optional projects follow, but these times will vary significantly depending on your students:
  • 40-60 minutes (1-2 class periods) on Astrid Design
  • 90-120 minutes (2-3 class periods) on Tic-Tac-Toe With An Intelligent Computer Player

Lab Pages

  • Page 1: Astrid Design.
    • Learning Goals:
      • Apply skills at developing algorithms and using abstractions in the context of a visual arts project.
      • Gain dexterity in the field of computational art.
      • Synthesize artistic sensibilities and programming instincts.
    • Tips:
      • Some students, looking ahead at the Take It Further exercises, may try to use too much abstraction in their initial construction of the Astrid and get frustrated at their lack of progress. The task given in the For You To Do exercises requires only joining points with fixed coordinates and should be relatively straightforward to do with the use of a for loop. Encourage them not to look too far ahead into the Take It Further activities which involve significantly harder tasks.
      • Some students will find it difficult to generalize the algorithm they developed to generate the first quadrant Astrid and apply it to other quadrants. It might be helpful to bring to their attention that first quadrant Astrid involves joining positive y-axis points to positive x-axis points, while the second quadrant Astrid involves joining positive y-axis points to negative x-axis points, etc.
  • Page 2: Tic-Tac-Toe With An Intelligent Computer Player.
    • Learning Goals:
      • Practice using higher order functions in an extended project.
      • Make the shift from the student's intuitive, informal tic-tac-toe strategy to an explicit, formal, algorithmic set of rules.
    • Tips:
      • Read the solutions linked below, and be sure you understand how higher order functions are used in each exercise, before class.
      • (2017â€“18 only) This project builds on the one in Unit 3, but students must start with the starter project provided for them, rather than from their own U3 solution, because (as the student page explains) this project requires a small change to the data structures they used in U3. (After this year, the U3 lab will be revised so that students can use their own solution as the starting point here.)
      • Students must alternate between two different points of view: (1) thinking about tic-tac-toe as a game, articulating the rules they use themselves when playing; (2) thinking about the problem of embodying those rules in a computer program. We try to make this explicit on the page, so, for example, FYTD 1-2 are about the program; FYTD 3-5 are about the game strategy; 6-13 are about the program; TIF A is about strategy; TIF B is about programming. But be on the lookout for students who confuse these two levels of abstraction.
      • FYTD 2 is at the heart of the actual program; consider having a whole-class discussion about it. Board is the user interface's representation of who has moved where on the board, a straightforward list of nine squares, each of which can be X, O, or the square number to indicate empty. Status of winning triples represents the same board status, so in a sense it's redundant, but it presents the board status in a form that's "tuned up" for the the choice of moves for the computer. You could say that board is for talking, while status of winning triples is for thinking.
      • In particular, students must understand the idea of a triple in this program. It's not any set of three squares; it's a set of three squares that count as a win: a row, a column, or a diagonal. The procedure TicTacToe wins, which reports a list containing the eight possible winning triples, is the representation in the program of the rules of tic-tac-toe.
      • In the first three units, students learned about three specific higher order functions: map, keep, and combine. Those three are very common in functional programming, but here we provide a fourth, find first, that happens to be useful in several places in this program. Make sure that students notice this; later in the course, students will learn to design and write their own higher order functions.
      • If students have been told that a computer program is "a sequence of steps," it may take some intellectual effort to learn to think instead of a program as a composition of functions. But the effort pays off in vastly reducing the work of writing the actual code. For example, students are asked to write winning square for, a function whose input is "X" or "O" and whose return value is a number from 1 to 9, or 0 if there is no winning square on this move. Some students will want to jump right into coding, and will have a very long sequence of if board=(some list) report (some number). (There are 39=19,683 possible board configurations, so it would be a very long procedure if they try to cover every one.) But finding a winning triple is just find first from the list of winning triples, where the predicate is, e.g., "this triple has two O squares and no X squares." And then the actual winning square is find first looking for a number in the winning triple. The resulting program isn't quite a one-liner because it has to check for the case in which there is no winning triple, but it's hardly any code compared to the sequence of if statements.
      • TIFs A and B are reasonable extensions to this program, but TIFs C and D suggest completely different approaches to writing a tic-tac-toe player. Even students who always do the TIFs might want to defer these until after the AP exam.
