Programming Lab 1: Contacts List

Lists can contain anything: numbers, strings, other lists, even blocks. The objects in a list can be all of the same kind (all numbers, for example, or all lists that contain a name and a phone number) but they do not have to be. A list can, for example, contain an apartment number and a list of people at that address. The tabular data we are familiar with in spreadsheets can be represented as a list of same-length lists, the smaller lists representing each row in the table, and the enclosing list representing the entire spreadsheet.

The "ordered pairs" of algebra, the (x, y) coordinates of a point, can be represented as lists. So can vectors and matrices. The list block is the basic building block of these complex structures. To select information in the dataset, we only need to know where it appears.

Blocks can locate items in the nested lists. For example, item (3) of the list of points below gives {10, 120}. As the sprite visits each of these points (in a for each block) the point (10, 120) is the third that it visits.

Students start by constructing a very simple contact list app. The list's elements are themselves lists. This requires not only collecting and storing information, but being able to retrieve it in useful ways. Students use keep to be able to collect only the information they need from a list, and learn to use for each to process lists of points on the plane—points which are themselves lists of coordinates. Data Types, a later page in this lab, returns to the idea of using abstraction to make the (x, y) coordinate lists more recognizable and usable by creating the abstract data type point and constructing and identifying these coordinate lists that way. After some embellishment of the contact app, practicing and extending the ideas of abstraction, and a Lists Review and Self Check, students extend their use of for each and data types to a musical context.

Thus in Labs 1 and 2, students have encountered lists of strings/words, numbers, and lists. They also use abstraction to clarify and simplify code, and see how it makes code more flexible and maintainable.

Students will:

  • Use the keep function to search for a contact in the Contact List app.
  • Use the for each block to access each item of a list.
  • Create abstract data types to organize and manage data.
  • Use lists of lists together with data types to organize and access data.

Assignments:

  • Assessment Idea: Write a program that keeps track of each student's grade in one class.
    1. Create abstract data type to hold each student record:
      • Create a student record constructor—a reporter block that, given a name (a text string) and grade (a number) as input, reports a student record (a list of that name and grade).
      • Create two selectors name and grade that, given a student record as input, report the name or grade for that student record.
    2. Make a variable class that, when the green flag is clicked, is set to an empty list that will store student records.
    3. Add a button (sprite) that uses your constructor to allow the user to create a student record by entering the student’s name and grade.
    4. Add a button that generates a grade report. Use the for each block to go through each record in class and have the sprite say the student’s name and grade for a second. Format the output so that it makes sense to the user.
    5. Submit your code in whatever way your class submits work.

Tips:

  • Many of the snap projects linked from the student pages already have the tools imported.
    Students need to each time they open a new snap window. (Links to projects in the curriculum already contain the tools.)
  • Page 1: Contact List App.
    • The Contact List project is ambitious but carefully scaffolded. It's helpful for you to remember its ultimate goal, which is to introduce the idea of data abstraction, in which an abstract data type is defined by writing constructor and selector blocks that are the sole program interface to the underlying list structure. We start without data abstraction, so you see lots of item 2 of when a phone number is wanted. By the end of the project, there should be no more item calls in the user interface blocks. Students will be inclined to resist data abstraction, because it makes the program longer. But the project uses a change in the underlying representation to convince them that it's much easier to maintain a large program if such changes affect only the constructor and selectors.
  • Page 2: The For Each Block.
    • Encourage students to first complete the two blanks in the script (shown at right) and only after most of the letter is drawing correctly to begin addressing the two bugs at the beginning and end of the drawing process.
    • For the Take It Further problem, students may want/be asked to review the concept of slope from algebra to identify the remaining points in the letter.

    • Debugging: As students fix the bugs in Morgan and Betty's script, they must seek the cause(s) of the unwanted and missing lines in the drawing and identify the appropriate solution.
    • Mathematics Note: Coordinates Again. The list A contains the coordinates of the special points (vertices) that define the picture of the letter A in that position on the stage. You can use these coordinates to draw an exact replica of A in another spot on the stage. This would be useful, for example, if you want to set two letters next to each other on the screen, or make a "shadow" of A or to draw an offset image to create a three-dimensional effect. Try creating a list AA from A by adding, say, 50 to each of the coordinates of each of the points, and then plot the resulting points with for each. Later in this unit, you'll see how to produce AA with a lot less trouble, using map.
  • Page 3: Data Types.
    • "Abstract data type" (or ADT) sounds like a complicated thing, but programmers actually use them to simplify and clarify the logical structure of their code. It is a thing to help people, not machines. An abstract data type is like a "template" for data—for example, a registration form on which individuals write their name, address, email, etc., in a particular place helps us know where to look for each part of the data.
    • Mathematics Note: Abstraction. You can make the script even easier to read and understand if you create a block glide to pt, made from glide, that glides to a point rather than to a pair of coordinates.
  • Page 4: Improving Your Contact List App (Teacher's Choice).
    • This is a longer page, and it's possible for students to lose the big picture among all the details. You should understand the following structure of the three green For You To Do boxes:
      • The first box breaks the app by making a change to its data structure. Don't let students get discouraged by this; it'll work out fine in the end. The moral of this first part is that programs can be brittle when any part of the program changes.
      • The second box makes the program less brittle by formally defining a Contact abstract data type (ADT).
        • Such a definition consists of one (occasionally more) constructor block, and several selector blocks. Constructors create an instance of the data type; selectors pull out pieces from such an instance. Once the ADT is set up and is consistently used by the rest of the program, it's possible to change the internal representation without breaking anything other than the constructor and selectors.
        • If students think this is too much work for the benefit, ask them to imagine a real-world program with thousands of blocks (procedures) and dozens of data types. Then imagine the situation in the first green box: You need to move the phone number from item 2 to item 3 of a contact. Without data abstraction, you'd have to scan through the entire program looking for item 2 of expressions, and for each one you find, you have to work out whether it is looking at a contact (in which case you should change the item number) or at something else represented by a list (in which case you shouldn't). With data abstraction, there are no item calls outside of the selectors; the rest of the program should keep working smoothly when the representation changes.
      • The third For You To Do box gives the payoff for this work. We now want two kinds of contacts, ones with the family name first and one with the given names first. We make a major change to the constructors (two of them, now) and selectors, but the rest of the program still works, better than ever.
  • Page 5: Lists Review and Self Check (Teacher's Choice).
    • Students familiar with music may want to use fractions (e.g. 0.25, 0.5, etc.) to represent note lengths (e.g. quarter notes, half notes, etc.) and adjust their tempo(s) accordingly. Integer note lengths are offered on the lab pages for students unfamiliar with the nuances of representing musical time signatures as decimals (e.g. 0.25 + 0.5 being a full measure in 3/4 time, but not 4/4 time). Integers work just as well and are more forgiving.

  • EK 2.1.1A Digital data is represented by abstractions at different levels.
  • EK 2.2.1A The process of developing an abstraction involves removing detail and generalizing functionality.
  • EK 2.2.1B An abstraction extracts common features from specific examples in order to generalize concepts.
  • EK 2.2.1C An abstraction generalizes functionality with input parameters that allow software reuse.
  • EK 2.2.2A Software is developed using multiple levels of abstractions, such as constants, expressions, statements, procedures, and libraries.
  • EK 2.2.2B Being aware of and using multiple levels of abstraction in developing programs helps to more effectively apply available resources and tools to solve problems.
  • EK 2.2.3D In an abstraction hierarchy, higher levels of abstraction (the most general concepts) would be placed toward the top and lower level abstractions (the more specific concepts) toward the bottom.
