Programming Lab 1: Introduction to Lists

Computers manipulate information they are given and return processed information to the user. Numbers, strings of characters (for example, words), and Boolean values (true, false) are some of the basic pieces of information. Often these small pieces are arranged in more complex structures: trees of information, lists (in mathematics, vectors), tables or arrays (in mathematics, matrices), and more. The list is the basic building block of all those more complex structures. A table of data, for example, is a list of rows of data, each row being a list of data. A list is not just a collection (set) but a collection in a particular order. If it has any elements at all, it has a first element, a last element, and so on. You can search a list to see if it contains some element; you can create new lists that keep only certain items from a list you started with; you can create new lists with more or fewer elements than the list you have; and you can pick an item of a list by position, or characteristic, or just randomly. A list can be totally empty, with no elements at all; often that is the way we want to start (initialize) a list before we add elements to it. The length of such a list is 0. An item (e.g., almonds) is not the same as a list containing just that item. And you can do something with each item of the list: you can process it in some way (e.g., map a function over each element and report the resulting list) or, for each element of the list, you can display it in some way.

Students begin Lab 1 by building a very elementary shopping list app: it its simplest form, the app allows the user only to add items to the list or clear the list. Students can later add more features as they learn more about list processing. Students also process lists of points on the plane and lists of notes in a melody. The points and notes are, themselves list of two numbers, in one case the list (x, y), and in the other case the list (pitch, duration). Thus, students, in Lab 1, have encountered lists of strings/words, numbers, and lists. They also use abstraction to clarify and simplify code, clarifying the role of a pair of numbers as the coordinates of a point (or the attributes of a note) not just as the elements of a list.

  • Exploring List Blocks:
    • This page asks students to build the blocks (rather than providing a link that has the blocks built already). This is intentional, so that students get the experience of finding these blocks in the palette and getting the logic of what kinds of blocks go inside of other blocks.
    • Students explore blocks such as all but the first of {}, () in front of {}, and item () of {}. Consider asking students to write out a sentence describing the behavior of each of these blocks or to describe them to the class
    • Listen for students making the distinction between expressions that report strings (such as "Boise") and those that report a list containing one item (such as {Boise}). If you don't hear students making this distinction, consider a brief class discussion about it. Distinguishing the logic between what data type is reported is a big help when debugging.
