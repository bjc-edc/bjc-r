Programming Lab 2: Nesting Lists

A list of same-size lists forms a table, a two-dimensional array of data. Their ordered and nested indicies provide a logical structure for accessing information in the dataset. (In mathematics, lists appear as vectors and lists of lists as matrices.)

, tables or arrays of data (in mathematics, vectors and matrices), and more. The list is the basic building block of all those more complex structures. A table of data, is a list of rows of data, each row being a list of data.

Students create a contact list app that collects and stores phone numbers and names and can be cleared, and explore list reporters with nested lists. They use for each to process lists of points on the plane and lists of note pitches and durations in a melody. The points and notes are, themselves list of lists, ({x, y} and {pitch, duration}).

abstract data types (ADTs)

Thus in Labs 1 and 2, students have encountered lists of strings/words, numbers, and lists. They also use abstraction to clarify and simplify code, clarifying the role of a pair of numbers as the coordinates of a point (or the attributes of a note) not just as the elements of a list.

Lists that appear as elements of another list are called "nested lists". For example if we item (3) of the list below gives list {10, 120}. The point (10, 120) is the third that the sprite visits and is the first on the right hand (positive) side of the vertical axis. blocks can locate items in the lists in the list and also the just the lists within the list with all their items inside. Lists can go inside lists inside lists and so on, but Unit 3 only covers two-dimensions of nesting: lists of lists which form arrays. in is a list that appears as an element in another list. In this list, the element with index 3 is a nested list. To extract an element from the nested list, we can proceed in two steps. First, extract the nested list, then extract the item of interest. It is also possible to combine those steps using bracket operators that evaluate from left to right.

As a Class:

  • Page 1: Lists of Lists.
    • Discussion, you might ask:
      1. What is a list of lists?
      2. What kinds of things can you do with a list of lists?
      3. Do the lists inside a list need to be the same size?
      4. What can be done if the lists inside a list are the same size?
      5. In what other situations might a list of lists be helpful?
    • You might consider asking students (or pairs) to write a sentence for each of the list reporters decribing how they treat lists of lists. Or you could discuss this as a class as well.
  • Page 2: The For Each Block.
    • Encourage students to read though the dialogue more than once: individually, in small groups, in front of the class, etc.
  • Page 3: Data Types.
    • Encourage students to read though the dialogue more than once.
    • Discussion: After all students have read through the dialogue a few times (and perhaps also added the abstract data type to their U3L1-DrawShape project), you might review with questions like these:
      • What is the benefit of creating an abstract data type?
      • In what other situations might abstract data types be helpful?
      • What kind of data types does a list take? Lists take any type of input (primative or abstract data types)
      • What is a primitive data type? What are some examples?
      • What is a abstract data type? What are some examples?
      • What is the difference between a primitive data type and an abstract data type?
      • How are abstract data types an example of abstraction?
  • Page 4: Lists Review and Self Check.
    • Consider inviting students with musical training to present basic music theory (especially classical note pitch and beat length representation) and to share and describe their music script(s) with the class.
