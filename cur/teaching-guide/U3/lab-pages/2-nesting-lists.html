Programming Lab 2: Nesting Lists

A list of same-size lists forms a table, a two-dimensional array of data. The ordered and nested indicies of the lists provide a logical structure for accessing information in the dataset. In mathematics, lists appear as vectors and lists of lists as matrices. The list is the basic building block of these more complex structures.

Lists that appear as elements of another list are called "nested lists". Blocks can locate items in the nested lists and also the nexted lists themselves (with all their items inside). For example, item (3) of the list of points below gives list {10, 120}. As the sprite visits each of these points (in a for each block) the point (10, 120) is the third that it visits.

Lists can go inside lists inside lists and so on, but Unit 3 only covers two-dimensions of nesting: lists of lists which form arrays.

Students begin the Lab on page 1, Lists of Lists, where they explore list reporters with nested lists and create a contact list app that collects and stores phone numbers and names. Then on page 2, The For Each Block, they use for each to process lists of points on the plane—points which are themselves lists. Page 3, Data Types, introduces the idea of using abstraction to make the lists inside the list more usable, namely, labeling them as points. Finally in page 4, Lists Review and Self Check students use for each two write two additional sprites (Show Contact and Delete Contact) for their Contact List app and extend their use of for each and data types to a musical context.

Thus in Labs 1 and 2, students have encountered lists of strings/words, numbers, and lists. They also use abstraction to clarify and simplify code, clarifying the role of a pair of numbers as the coordinates of a point (or the attributes of a note) not just as the elements of a list.

Pacing:

  • The four pages of this lab are each expected to be relatively quick, approximately 1 class period each.

Prepare:

  • Page 1, Lists of Lists, and Page 4, Lists Review and Self Check include a new Contact List app that was not part of this curriculum until October 2015. Teachers should review and work though these pages. Feedback is always appreciated.

Students will:

  • Access and manipiulate data in lists of lists.
  • Use the for each block to access each item of a list.
  • Create abstract data types to organize and manage data.
  • Use lists of lists together with date types to organize and access data for graphics and music scripts.

As a Class:

  • Page 1: Lists of Lists.
    • Discussion, you might ask:
      • What is a list of lists?
      • What kinds of things can you do with a list of lists?
      • Do the lists inside a list need to be the same size?
      • What can be done if the lists inside a list are the same size?
      • In what other situations might a list of lists be helpful?
    • You might consider asking students (or pairs) to write a sentence for each of the list reporters decribing how they treat lists of lists. Or you could discuss this as a class as well.
  • Page 2: The For Each Block.
    • Encourage students to read though the dialogue more than once: individually, in small groups, in front of the class, etc.
  • Page 3: Data Types.
    • Encourage students to read though the dialogue more than once.
    • Discussion: After all students have read through the dialogue a few times (and perhaps also added the abstract data type to their U3L1-DrawShape project), you might review with questions like these:
      • What is the benefit of creating an abstract data type?
      • In what other situations might abstract data types be helpful?
      • What kind of data types does a list take? Lists take any type of input (primative or abstract data types)
      • What is a primitive data type? What are some examples?
      • What is a abstract data type? What are some examples?
      • What is the difference between a primitive data type and an abstract data type?
      • How are abstract data types an example of abstraction?
  • Page 4: Lists Review and Self Check.
    • Consider inviting students with musical training to present basic music theory (especially classical note pitch and beat length representation) and to share and describe their music script(s) with the class.

  • Page 3: Data Types.
    • "Abstract data type" sounds like a complicated thing, but programmers actually use them to simplify the logical structure of their code. It is reasonable to think of an abstract data type like a "template" for data—for example, if you had everyone in class fill out a form with their name, address, email, etc., that form would be like an abstract data type (except for being a real paper form—not an abstract digital artifact).
  • Page 4: Lists Review and Self Check.
    • Students familiar with music may want to use fractions (e.g. 0.25, 0.5, etc.) to represent note lengths (e.g. quarter notes, half notes, etc.) and adjust their tempo(s) accordingly. Integer note lengths are offered on the lab pages for students unfamiliar with the nuances of representing musical time signatures as decimals (e.g. 0.25 + 0.5 being a full measure in 3/4 time, but not 4/4 time). Integers work just as well and are more forgiving.
