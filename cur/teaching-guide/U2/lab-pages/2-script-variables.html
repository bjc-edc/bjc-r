Programming Lab 2: Script Variables

Students have already created blocks that have input variables, and have used those within the block's script. This lab introduces the block, which allows the programmer to create variables for scripts that are not inside blocks, and extra temporary variables for blocks that need them. They allow you to save, use, or modify information that the script will need while it is running. Script variables exist only while the script is running and only inside that script. Blocks used inside a script can set, use, or change the value of that script's variables; blocks outside of that script can not.

Students get a choice of small projects in art or language or mathematics and get plenty of practice building blocks of various types including predicates. Students get continued opportunities to debug before lab 3, which begins to focus on the development of more complex programs.

Prepare:

  • Getting, renaming, setting, and using a script variable are all new. Some students will need help just building the script shown in problem 1 on the first page of this lab. An animation shows most of what they will need to do, but make sure you are comfortable enough, yourself, to help if that is needed.
  • With other problems, it can actually be useful for you not to look fully fluent, even if you know exactly what to do. Showing how you puzzle out the problem can be an excellent way of modeling the thinking for your students. You don't have to "fake" not knowing. Your genuine curiosity about what the student has already done is likely to be enough. Ask what various bits of the program do, or what they're supposed to do, or how the student figured out what to put in or where to put it. You experiment with the student's program, pausing for a moment to think about what it did, or suggest trying various inputs to see how the program responds. These can be enormously helpful to the student because they illustrate a good analysis/debugging technique. Often enough, students then find the solutions themselves.

Tips:

  • Be aware that the curriculum deliberately starts from variables with limited scope (the variable in the for block, custom block inputs, and now script variables), giving students as much experience as possible with them before introducing global variables. The goal is to get students in the strong habit of using local variables long before global variables are introduced. At this point, you don't need to (and shouldn't) talk about scope issues or global vs. local variables at all.
  • Students who choose the "writing secret code" project will encounter unicode and explore it a bit. They can be leaders in discussions of that topic when it comes up later in the year.
  • It's possible to get a non-keyboard character into the unicode of block by copying it in another program (from a web page or a text file, for example) and then pasting it in Snap! with control-V or command-V.
  • This is a perfect place for the creativity called for in the CS Principles. Try to find time for students to share the different ideas they've brought to their projects.
  • At the end of the Script Variable Projects page is a teaser for the functional programming style coming in Unit 3. As always when we foreshadow later parts of the curriculum, let it remain just a taste at this point, without calling special attention to it.
  • In the Building a useful collection of predicates page, many students find the first problem (>=) difficult. Encourage them not to give up. This taks seems to lead to a big improvement in students' understanding both of logic and of composition of functions. Later in this unit there is a student page entirely about this exercise, comparing different solutions. Don't have the students jump ahead, but read it yourself to get a sense for different kinds of solutions to look for. If you get different ones, have students explain their work to the class.
  • Students sometimes type the words "true" and "false" into report blocks instead of using the Boolean constant blocks. That bug can take hours to find and isn't intellectually interesting. If you notice it, just tell them how to fix it so that they have time to get to the more interesting logic bugs.
  • In the Analyzing, debugging, and simplifying code page, take note of the last question, 4c. The point it's making is important: When computing a function, write a reporter, not a command block that says the value. The latter can't participate in composition of functions. In the case of max, we're later going to compose it with itself to find the maximum of three numbers. People who've programmed in non-interactive languages aren't accustomed to being able to click on a reporter to see what it reports, and tend to write scripts with lots of ask and wait blocks at the beginning and lots of say blocks at the end. Instead, write a reporter with inputs.
