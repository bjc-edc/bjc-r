Lab 1: Improving Games by Using Variables

In this lab, students build a number-guessing game in which the player tries to guess the computer's secret number, and the computer responds with as human-like dialogue as the programmer can build. This project reviews conditionals and predicates before students refine their understanding of predicates and use them with keep in the following lab, and it also introduces the topic of script variables (a.k.a. local variables) for the storage of the computer's secret number.

Students have already encountered variables in two contexts: they have created blocks with input variables, and they have used the for block, which provides a index variable (i.e., a counter). This lab introduces the block, which lets programmers create their own temporary variables, in which to store, use, or modify information that a script will need while it is running. Script variables exist only while the script is running and only inside that script. Inside that script, blocks can set, use, or change the value of that script's variables, but blocks outside that script cannot.

Students take time to debug each portion of their project before adding each next page. They are given suggestions for features to add, one at a time, but are on their own to think about how to structure these extensions and modifications so that the code works and remains clear.

Pacing

The 4 lab pages could be split across 3-5 days (100-200 minutes). Expected times to complete follow:
  • 30-60 minutes (about 1 class period) on Developing a Number Guessing Game with Script Variables
  • 30-60 minutes (about 1 class period) on Debugging and Extending Your Number Guessing Game
  • 20-40 minutes (about 1 class period) on Keeping Score with Global Variables
  • 20-40 minutes (about 1 class period) on Using Greet Player in Another Program

Lab Pages

  • Page 1: Developing a Number Guessing Game with Script Variables.
    • Goals:
      • Create and use a script variable to store temporary data.
      • Use the repeat until conditional to control program flow.
      • Use the ask and answer blocks to manage text input.
    • As a Class:
      • There is an introductory offline activity suggested for this page included in the Engaging Students in Building Algorithms Offline section of the introduction to this Teacher Guide.
  • Page 2: Debugging and Extending Your Number Guessing Game.
    • Goals:
      • Analyze and debug existing code.
      • Implement a second script variable.
      • Implement additional conditionals with predicates.
    • Tips:
      • Debugging is a skill to refine over time. To support its development, analyze and debug projects together as a class. You may also have students share bugs they have encountered either to solicit suggestions from the class or to share a successful debugging approach.
  • Page 3: Keeping Score with Global Variables.
    • Goals:
      • Create and use a global variable to keep track of a game score.
      • Use conditionals to end the game when the player reaches a target score.
      • Use stop blocks to manage program flow.
    • Tips:
      • Be sure that students rename their project with a Unit 2 filename: U2L1-ClickAlonzo as Snap! will overwrite their Unit 1 project if students save their work under the U1L1-ClickAlonzo filename.
  • Page 4: Using Greet Player in Another Program.
    • Goals:
      • Export blocks and import them successfully into other projects.
      • Debug possible dependencies on global variables

Correlation with AP CS Principles Framework: 

Enduring Understandings:

  • EU 1.1 Creative development can be an essential process for creating computational artifacts.
  • EU 2.2 Multiple levels of abstraction are used to write programs or create other computational artifacts.
  • EU 5.3 Programming is facilitated by appropriate abstractions.

Learning Objectives:

  • LO 1.1.1 Apply a creative development process when creating computational artifacts. [P2]
  • LO 2.2.1 Develop an abstraction when writing a program or creating other computational artifacts. [P2]
  • LO 2.2.2 Use multiple levels of abstraction to write programs. [P3]
  • LO 5.3.1 Use abstraction to manage complexity in programs. [P3]

Essential Knowledge:

  • EK 2.2.1A The process of developing an abstraction involves removing detail and generalizing functionality.
  • EK 2.2.1B An abstraction extracts common features from specific examples in order to generalize concepts.
  • EK 2.2.1C An abstraction generalizes functionality with input parameters that allow software reuse.
  • EK 2.2.2A Software is developed using multiple levels of abstractions, such as constants, expressions, statements, procedures, and libraries.
  • EK 5.3.1A Procedures are reusable programming abstractions.
  • EK 5.3.1B A procedure is a named grouping of programming instructions.
  • EK 5.3.1C Procedures reduce the complexity of writing and maintaining programs.
  • EK 5.3.1D Procedures have names and may have parameters and return values.
  • EK 5.3.1E Parameterization can generalize a specific solution.
  • EK 5.3.1F Parameters generalize a solution by allowing a procedure to be used instead of duplicated code.
  • EK 5.3.1G Parameters provide different values as input to procedures when they are called in a program.
  • EK 5.4.1C Meaningful names for variables and procedures help people better understand programs.
