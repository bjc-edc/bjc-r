Lab 2: Making Art by Using Data Structures

Lists can contain anything: numbers, strings, other lists, even blocks. The objects in a list can be all of the same kind (all numbers, for example, or all lists that contain a name and a phone number) but they do not have to be. A list can, for example, contain an apartment number and a list of people at that address. The tabular data we are familiar with in spreadsheets can be represented as a list of same-length lists, the smaller lists representing each row in the table, and the enclosing list representing the entire spreadsheet.

The "ordered pairs" of algebra, the (x, y) coordinates of a point, can be represented as lists. So can vectors and matrices. The list block is the basic building block of these complex structures. To select information in the dataset, we only need to know where it appears.

Blocks can locate items in the nested lists. For example, item (3) of the list of points below gives {10, 120}. As the sprite visits each of these points (in a for each block) the point (10, 120) is the third that it visits.

Students start by constructing a very simple contact list app. The list's elements are themselves lists. This requires not only collecting and storing information, but being able to retrieve it in useful ways. Students use keep to be able to collect only the information they need from a list, and learn to use for each to process lists of points on the plane—points which are themselves lists of coordinates. Data Types, a later page in this lab, returns to the idea of using abstraction to make the (x, y) coordinate lists more recognizable and usable by creating the abstract data type point and constructing and identifying these coordinate lists that way. After some embellishment of the contact app, practicing and extending the ideas of abstraction, and a Lists Review and Self Check, students extend their use of for each and data types to a musical context.

Thus in Labs 1 and 2, students have encountered lists of strings/words, numbers, and lists. They also use abstraction to clarify and simplify code, and see how it makes code more flexible and maintainable.

  • Page 2: Data Types.
    • Goals:
      • Create abstract data types to organize and manage data.
      • Use lists of lists together with data types to organize and access data.
      • Understand what a data type is.
      • Understand how data types help manage program complexity and increase clarity, especially for testing, debugging and maintenance.
    • Tips:
      • Again, consider having students read the dialogue more than once.
      • Discussion: After all students have worked through the dialogue (and perhaps also added the abstract data type to their U3L2-DrawShape project), you might review with questions like these:
        • What are some benefits of creating an abstract data type?
        • In what other situations might abstract data types be helpful?
        • What is a primitive data type? What are some examples?
        • What is an abstract data type? What are some examples?
        • What is the difference between a primitive data type and an abstract data type?
        • How are abstract data types an example of abstraction?
      • "Abstract data type" (or ADT) sounds like a complicated thing, but programmers actually use them to simplify and clarify the logical structure of their code. It is a thing to help people, not machines. An abstract data type is like a "template" for data—for example, a registration form on which individuals write their name, address, email, etc., in a particular place helps us know where to look for each part of the data.
      • Mathematics Note: Abstraction. You can make the script even easier to read and understand if you create a block glide to pt, made from glide, that glides to a point rather than to a pair of coordinates.
  • Page 3: Mandala Design.
    • Goals:
      • Apply everything learned in this lab in an artistic context.
      • Use coding as a tool for creative and artistic expression.

    • Tips:
      • This page relies on the succesful mastery of the last two pages, so it is essential that the students do not attempt this work until they have reached a level of comfort with the earlier concepts.

Assignments

  • Assessment Idea: Write a program that keeps track of each student's grade in one class.
    1. Create abstract data type to hold each student record:
      • Create a student record constructor—a reporter block that, given a name (a text string) and grade (a number) as input, reports a student record (a list of that name and grade).
      • Create two selectors name and grade that, given a student record as input, report the name or grade for that student record.
    2. Make a variable class that, when the green flag is clicked, is set to an empty list that will store student records.
    3. Add a button (sprite) that uses your constructor to allow the user to create a student record by entering the student’s name and grade.
    4. Add a button that generates a grade report. Use the for each block to go through each record in class and have the sprite say the student’s name and grade for a second. Format the output so that it makes sense to the user.
    5. Submit your code in whatever way your class submits work.

Enduring Understandings:

  • EU 2.2 Multiple levels of abstraction are used to write programs or create other computational artifacts.

Learning Objectives:

  • LO 2.2.1 Develop an abstraction when writing a program or creating other computational artifacts. [P2]
  • LO 2.2.2 Use multiple levels of abstraction to write programs. [P3]
  • LO 2.2.3 Identify multiple levels of abstractions that are used when writing programs. [P3]

Essential Knowledge:

  • EK 2.1.1A Digital data is represented by abstractions at different levels.
  • EK 2.2.1A The process of developing an abstraction involves removing detail and generalizing functionality.
  • EK 2.2.1B An abstraction extracts common features from specific examples in order to generalize concepts.
  • EK 2.2.1C An abstraction generalizes functionality with input parameters that allow software reuse.
  • EK 2.2.2A Software is developed using multiple levels of abstractions, such as constants, expressions, statements, procedures, and libraries.
  • EK 2.2.2B Being aware of and using multiple levels of abstraction in developing programs helps to more effectively apply available resources and tools to solve problems.
  • EK 2.2.3D In an abstraction hierarchy, higher levels of abstraction (the most general concepts) would be placed toward the top and lower level abstractions (the more specific concepts) toward the bottom.
  • EK 5.1.2G Program development includes identifying programmer and user concerns that affect the solution to problems.
  • EK 5.1.2J A programmer designs, implements, tests, debugs, and maintains programs when solving problems.
  • EK 5.1.3B Collaboration facilitates multiple perspectives in developing ideas for solving problems by programming.
  • EK 5.1.3F Effective communication between participants is required for successful collaboration when developing programs.
  • EK 5.2.1C Program instructions may involve variables that are initialized and updated, read, and written.
  • EK 5.3.1I Strings and string operations, including concatenation and some form of substring, are common in many programs.
  • EK 5.3.1K Lists and list operations, such as add, remove, and search, are common in many programs.
  • EK 5.3.1L Using lists and procedures as abstractions in programming can result in programs that are easier to develop and maintain.
  • EK 5.5.1H Computational methods may use lists and collections to solve problems.
  • EK 5.5.1J Basic operations on collections include adding elements, removing elements, iterating over all elements, and determining whether an element is in a collection.
