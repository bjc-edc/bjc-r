This lab introduces two small and extensible projects programming the computer to produce parts of English phrases: a block that can decide which of "a" or "an" to use with the following noun and a block that spells the correct plurals of input nouns. Both of these projects require conditionals (students focus on if else and if) using predicates (blocks that report true or false based on tests they perform) to control the behavior of their programs. Students use built-in predicates such as < and =, and they learn to build their own predicates, such as vowel?, which reports true if its input letter is a vowel and false if it is not. Students also use conditionals to control action based on the results of predicates that test the direction and y position of the sprite.

One way to introduce the idea of conditionals to your students. When you go to an ATM machine, you touch buttons on its screen to indicate whether you want to deposit or withdraw. Here's how it "knows" what you want. Someone wrote a program that places those words—deposit and withdraw—somewhere on the screen. The program then tests to determine where you touch the screen or which button you press. The action the program takes next is conditional; it depends on what input you've given it with your touch. Another part of the program may test the size of the withdrawal you request and indicate if that is too large or not an amount it can give, or it may deliver that amount. Your smart phone's screen works a similar way. The program knows only where you touch, but because it also knows what word or picture it put in that location, it can "know what you want"—if you touch here, do one thing, else do something else. Distinguishing conditions and basing action on the distinction is central to all programming.

Small projects focus especially on using multiple if statements to distinguish multiple conditions. Students design a program that tests to see whether "red," "yellow," or "green" has been typed, and sets a traffic signal to that color; they design another program to decide when to add s and when to add es to make the plural of a noun.

This lab also re-emphasizes abstraction. Students analyze the complex task of correctly spelling plurals by breaking it into subtasks and creating blocks that specialize in these subtasks. They also practice reading and analyzing scripts they have not created, one particularly powerful way to learn to program similar behaviors. Debugging is an essential skill—especially as one's programs become more complex—but it takes time to acquire and hone that skill because there is no single formula or sub-skill for doing it well. This lab also begins to draw explicit attention to one aspect of debugging: deliberately trying to "break" a program, searching for ways that a user's input (or some other situation that arises) can cause a program to fail and accounting for those ways by testing for them and responding appropriately.

    • Mathematics Notes: Reporters. In snap, a reporter is block that returns a value. Why "reporter" instead of "function?"
      • One reason for this vocabulary is that in mathematics, a function is defined by its output for a given input. So, for example, the functions defined by x \mapsto 2(x+1) and x \mapsto 2x+2 are equal as functions. But they are defined by different algorithms, so we'd like to say that we have two reporters that define the same function.
      • In addition, we want to emphasize that reporters don't need to take a number as input and produce a number as output. (Neither do functions in mathematics, but many students will come to the course thinking that a function is always defined by an algebraic expression that accepts and returns numbers.) In this unit, an important kind of reporter is a predicate, which reports not a number but one of two values: true or false. An example is the operator <:
      • One more reason: Some reporters don't define functions at all. An example is pick random:


        Each time pick random 1 to 10 is called, it reports an integer between 1 and 10, and different calls will likely produce different results. So, pick random reports an output that is not uniquely defined for given inputs as is required of mathematical functions.

    • Mathematics Note: Lists. If you've plotted points on a coordinate grid, you've used the idea of an ordered pair of numbers. An ordered pair is an example of a list—in this case a piece of data that contains two items. In mathematics and in computer science, it's a useful abstraction to think of a list of objects as a single thing. List can be built up and manipulated in Snap!, using some primitive and user-built reporters. There are constructors, such as list, that let you build lists of any kinds of objects (including other lists) and there are selectors, such as item, that let you pull lists apart.

  • Page 4: Using Multiple If Statements.
    • If time is short, have students do only the first three problems on this page. There will be other opportunities to learn what they may skip here.
    • Mathematics Note: Absolute Value. There are several equivalent ways to define absolute value. The one here—the distance between a number and 0 on the number line—is all students need for this lesson. In mathematics class, absolute value is often thought of as a formula that treats positive and negative numbers differently. That formulation isn't needed here. It might, however, be interesting for algebra students to write a Snap! block abs that implements this "piecewise" definition:
      |x|=\begin{cases} x & \text{if } x \geq 0 \\ -x & \text{if } x < 0 \end{cases} Modeling this in Snap! can help students understand the mathematics.
  • Page 5: Developing a Block That Makes Plurals.
    • Students, even in high school, may have questions about what is the correct plural or correct way to spell it. Encourage them to check with each other, but also feel free to ask you. Nobody can possibly think of all possible words anyway, so let students know that it is equally okay to ignore words that they're not yet sure about. One side benefit of this project is that it helps students notice how much they do know and take for granted, and how much work it is for a computer to do what humans do naturally! Students may do this kind of activity in another language if they like. The Take It Further on page 4 suggests one such investigation.
  • Page 6: Plurals, Continued.
    • The following rule summaries are for your benefit. Don't teach them to students. The point is not for them to know the rules in advance but to discover them the way computational linguists do, by making a model, seeing if it works, and fixing it if it does not. You can contribute by debugging a specialist block they think they've finished by proposing a word that it might get wrong.
      • plural-h needs to recognize words that end with ch and sh so that it can join es to those. To all other words (for example, ones that end with th) we just join s. To do this, plural-h must look at the next-to-last letter of the word.
      • plural-y also needs to look at the next-to-last letter, in this case to see if it is a vowel. To words like day and boy with vowels before the y, we just join s. For the others, like spy and fly, we "change" the last letter (y) to i (by removing the y and using join to replace it with an i) and then join es.
