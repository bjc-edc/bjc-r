Programming Lab 4: Higher-Order Functions

We end the unit by what will seem like a step backwards, to relatively simple linear recursions. But this important lab has a hidden agenda: The simple examples can be generalized into the higher order functions of Unit 3. For students who've made it through the rest of Unit 7, this lab won't seem hard at all, but implementing higher order functions is a big idea that students can accomplish. The visual metaphors of Snap! enable this simple treatment.

Pacing:

The 3 lab pages should take 3 days (135-165 minutes). Expected times to complete follow:
  • 45-55 minutes (1 class period) on Plurals and Other Recursions
  • 45-55 minutes (1 class period) on Generalizing the Map Pattern
  • 45-55 minutes (1 class period) on Even Numbers and Filtering

Students will:

Looking Ahead
  • This is the last lab, so … summer?
  • Investigate useful code patterns that generalize.
  • Build map, keep, and combine.
  • Construct additional higher-order functions.

As a Class:

  • Page 1: Plurals and Other Recursions.
    • Discussion: What is the plurals code pattern useful for?
      • What is the base case?
      • How does the pattern ensure that the base case is reached?
      • How many recursive calls are needed?
      • Given a list, what sorts of things could this pattern be used for?
  • Page 2: Generalizing the Map Pattern.
    • Discussion: How does map work?
      • How is map similar to plurals? How is it different?
      • What is the base case?
      • How does map ensure that the base case is reached?
      • How does map use a function as input? What does the call block do?
  • Page 3: Even Numbers and Filtering.
    • Discussion: How does keep work?
      • How is keep similar to evens? How is it different?
      • What is the base case?
      • Why are there two different recursive calls?
      • How does keep ensure that the base case is reached?
      • What does keep use as input?

Tips:

  • Page 1: Plurals and Other Recursions.
    • Students may have a hard time playing along with the concept, so emphasize that you're focusing on a pattern of code.
    • If time permits, work with some more examples like squares as a full-class discussion. Try one or more of these examples, then ask the students for their ideas:
      • subtract 1 from each element of a list
      • round each element of a list to the nearest integer
      • converting Unicode numbers to letters
      Doing this may help students realize that the plurals code pattern is the same one that map uses.
    • The exaggerate block has a wide berth of options for students. Start by having students code the number-replacement piece, then let them work on whatever they want. If discussing this, the key is that the exaggerate helper function is called on each word, and the pattern recursively moves across the list of words. Note that since this pattern works on lists, the entire sentence must be converted to a list at the start, then converted back to a sentence at the end.
  • Page 2: Generalizing the Map Pattern.
    • Emphasize the similarity between the three pieces of code on the page, and what a generalization of that pattern might look like. At that point, students understand what map does, it is just a flexible version of the code pattern used by plurals.
    • The details of marking the input as a function and using the call block are worth going through, but ultimately the activity is about how map isn't very complicated. It's a generalization of the pattern used by a simple recursive reporter. If students get bogged by those details they may miss this important point.
    • Consider building pairup two ways; one way using map, and one way in the style of plurals with a helper function. Students may see through this work that the helper function is one input to map.
    • Interested groups may want to dig into the code of the internal map block, which is not much different than what is built here. Most of the additional code for the internal map block handles multiple lists as input to a single call, in order to process functions with multiple inputs. The mapone area of the block is the place to focus.
  • Page 3: Even Numbers and Filtering.
    • The page begins with recursive code for the evens block. If there is time, start by asking students to build this block, rather than reading the code written here.
    • Emphasize the similarity between evens, ends-e, and numbers, and what a generalization of that pattern might look like. At that point, students understand what keep does, it is just a flexible version of the code pattern used by evens.
    • Students may have difficulty constructing keep, since the details of building its inputs are unusual. Similar to map, an input type must be declared, but this time it is a predicate and not a reporter.
    • The work on combine and sort also allow for further opportunities to use the process that built map and keep. Emphasize to students the types involved, to help them decide when the gray-box input should be a reporter and when it should be a predicate.
    • If there is time, definitely dig into the code of the internal keep block. Its code is almost identical to the generalization of evens. Expect students' code for combine to also be very similar to the code of the internal block.
