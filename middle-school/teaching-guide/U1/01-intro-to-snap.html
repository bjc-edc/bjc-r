This lab serves as introduction to course. After a quick video from the curriculum developers, the class shares their experience and interest in computer science and then explores a simple offline activity designed to introduce the basics of how computers process data in functional programming. Students then create Snap! accounts and learn about the Snap! programming language, interface, and terminology (reporter blocks, input parameters) by using various blocks that greet the user in different ways and changing the input parameters to explore the impact on outputs.

In this course, the curriculum will provide students with starter projects for them to save to their accounts. It's also possible to create new projects, and students may wish to explore that on their own time.

Pacing

This lab is designed for 3–5 class periods (120–200 minutes).

  • Activity 1: BJC Culture and Norms 35–65 minutes
  • Activity 2: Draw Like a Computer approximately one class period
  • Activity 3: Creating a Snap! Account 15–30 minutes
  • Activity 4: Say Hello to Snap! 10–20 minutes

  • Learning Goals:
    • Understand the goals of the course.
    • Understand the growth mindset.
    • Establish classroom norms.
  • Activity Plan:
    1. After welcoming students to the class and addressing any administrative matters, share the BJC Middle School Introduction video. (5 minutes)
    2. Review the growth mindset by asking questions like the following: (5–10 minutes)
      • When is a time when you used the growth mindset? How did it help you?
      • When is a time you would have benefited from thinking with a growth mindset? what could you do differently in a situation like that next time?
    3. Guide students in collaboratively developing a list of classroom norms by asking questions like the following: (15–30 minutes)
      • What makes it easier for you to learn?
      • What gets in the way of your learning?
      • What classroom norms could we create to facilitate your learning?
    4. Lead a brief getting-to-know-you discussion by asking questions like the following: (10–20 minutes)
      • What is your computing experience?
      • What do you want out of this course?
  • We might want to include a feedback form for teachers to tell us what they learn about student programming experience and hopes for the class. --MF
    Tips:
    • Assure students that all levels of computing experience are welcome in this class. You are asking these questions so that you and the curriculum developers can tailor the class to the experience and interests of students.
    • If some students are hesitant to share their computing experience or there isn't enough time to hear from everyone, consider making it a brief (1 paragraph) writing assignment.
    • Make note of the types of responses students provide in the getting-to-know-you discussion so that you can share them with curriculum developers.
    • Allowing students to develop their own classroom norms gives them a sense of agency over classroom expectations. Use the suggested questions (or other probing questions) to help students consider how to create a classroom environment that maximizes learning and minimizes distraction and barriers to learning.

Activity 2: Draw Like a Computer. 

Awaiting a revision from Dan. We didn't do the Marching Orders because it was command/sequential focused, but the "translate a recipe to a flowchart" approach missed some of the goals of the activity. --MF, 7/9/21

  • Materials:
    • Each student will need several pieces of paper and a writing implement.
    • Draw Like a Computer student page
  • Prepare:
    • Watch Josh Darnit's Peanut Butter and Jelly and Drawing Exact Instructions Challenge videos to appreciate the spirit of the activities; we'll be doing the drawing activity in class today. We have done a similar CS Unplugged activity, called "Marching Orders" for years, and this is a re-imagining of it as a Functional-style activity.
    • As you can see, the key is that the person doing the drawing tries to mess up as much as possible while still following the directions specified to the letter. For example, if the programmer (the student speaking) says to draw a square, the computer (the student drawing) would draw a barely-imperceptible square (1mm on a side) since the programmer didn't specify how big it should be. Set up the activity as a game: the computer's job is to mess up as much as possible (while still following the "program"), and the programmer's job is to be as precise as possible.
  • Learning Goals:
    • Appreciating that you need precision to be able to describe something to a computer.
    • Appreciating that functions take inputs, process them in various ways, and produce an output.
  • Activity Plan:
    1. Explain the activity for the day -- describing the roles of programmer (read aloud a list of drawing instructions with the goal of having the computer replicate some original drawing) and computer (deviate as much as possible from the programmer's intentions, while following the commands precisely).  
    2. Ask the students to take a piece of paper out and serve the role of computer. Challenge them to see how far they can go 

      3. For example...

      Here's what might happen with the following instructions (note :

      • Draw a square.
      • Draw a circle on top of that.
      • Draw a triangle on top of that.

      Programmer's Original Drawing
      Possible Computer Drawings
    3. Students then create their own drawing (using a small set of geometric primitives) and write computer instructions for re-creating.
    4. In pairs, students take turns as "programmer", reading their list of commands while the other pretends to be a "computer", drawing the picture.
    5. Ask the students to come up with the picture that is supposed to be copied themselves, using only simple straight lines up down left right or 45° diagonal, circles, and arcs. Basically simple geometric sketches, nothing squiggly or anything. Those could all be collected and then reshuffled around the room in the standard marching orders way.
    6. Explain that the picture that they drew could also be thought of as coming from a function, like 2*(3+7). We’d show how that expression can be seen another way, as smaller one-task machines cascading into each other. I sometimes called the lower part of the picture below “funnel functions” because they look like funnels and return only one thing at the bottom and they are functions.
  • Tips:
    • This activity has been done for years as the Peanut Butter and Jelly activity, in which the teacher asks the students to describe how to make a PB&J sandwich as precisely as possible. As you see from the parent in the video, the key to the whole activity is that the parent is supposed to try to find ways to mess up.
    • Students may be inclined to write instructions that a computer couldn't follow (unless it were programmed to) such as draw a smiley face. Make sure students only write instructions using the list of commands provided.
    • Not dwelling on the fact that it’s mostly command-based.
  • Standards: None covered.

Activity 3: Creating a Snap! Account 

  • Materials: Creating a Snap! Account student page
    OLD: Creating a Snap! Account Google Doc
  • Prepare:
    • Before leading the activity with students, create your own Snap! account (if you haven't already) and customize your own settings.
    • You may find it helpful to review the beginning of the Beauty and Joy of Computing: Snap! Crash Course and perhaps also to familiarize yourself with the structure of the Snap! Reference Manual, which is accessible by clicking the Snap! logo in the top left corner of any Snap! window.
  • Learning Goals:
    • Create a Snap! account.
    • Learn to navigate the Snap! interface.
    • Customize the Snap! programming environment.
  • Activity Plan:
    1. Students create a Snap account.
    2. Students then customize several basic Snap! settings (language and appearance).
  • Tips:
    • During the signup process, new users are asked to click a link in their email to verify the account, but that doesn't have to happen right away.
    • Snap! users can change their password at any time using the Cloud menu.
    • Users can tell whether they are logged in by checking the Snap! Cloud menu:
      • If it says "Logout" and then a user name, then that user is logged in.
      • If it says "Login..." then no one is logged in.
      If students are using shared computers, they should log out of Snap! when they are done.
    • Make note of any questions or difficulties that arise, and please share them with the curriculum developers.
  • Standards:
    • None covered in this activity since it focuses on getting set up with Snap!.

Activity 4: Say Hello to Snap!. 

  • Materials: Say Hello to Snap! student page
  • Prepare:
    • Take a few minutes to explore the blocks in the Snap! project so you understand what they do before you share the Google doc with students.
  • Learning Goals:
    • Understand that a block is a piece of code that can be run by a computer and it can be run by clicking it.
    • Understand that reporter blocks are blocks that report outputs.
    • Understand that block can have inputs, those inputs can change the output, and the output of one block can be used as the input to another.
    • Explore changing inputs by typing different text into the input slots, by selecting different dropdown menu options, and by dragging other blocks into the input slots.
  • Activity Plan:
    • Students explore two blocks:
      • , which takes a name and a language as input and reports a greeting to that person in the selected language.
      • , which takes text as input and reports the same text with any letters changed to fancy Unicode letters
    • Students then nest the fancy block inside the say hello block and observe how the input of fancy becomes the input to say hello.
    • Finally, students learn to save their project to the Snap! cloud, as they should every day and periodically while working.
  • Tips:
    • In this first hands-on exploration with Snap!, students may struggle with basics such as clicking a block to make it run, difficulty typing into input slots etc. The simplicity of this project allows for debugging these simple challenges of the absolute beginner so that they are resolved by the time students move on to more complex use of the Snap! interface such as dragging blocks into each other, creating input parameters, and creating and editing blocks.
  • Standards:
    • None covered in this activity since it focuses on the basics of using Snap!.
