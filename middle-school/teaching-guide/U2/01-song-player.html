Lab 1: Song Player

After Unit 1 introduced programming in the functional style (using procedures that report values rather than change variables' values, and using procedures themselves as data), Unit 2 introduces imperative programming (using procedures that produce effects—sounds, movement on the Stage, and other changes to the state of the program). Lab 1 uses the sequential nature of music as a context for learning about iteration (repetition), visible stepping (a Snap! debugging tool), creating command blocks (blocks with effects), exploring the command versions of blocks of which students have already used reporter versions: if else and ask and wait (and its pair answer), and list traversal. Along the way, students review use of the higher order function map and gain informal experience with the abstract data type, ♪ (pronounced "note"), which contains both a pitch and a beat for any note in a song.

This curriculum focuses on giving students an engaging, strong foundation in programming without formalizing all ideas presented. Students don't need to learn this, but teachers may find it useful to know that an abstract data type (ADT) is a custom data type that's meaningful within a program.

An ADT is not built into the language (like common data types such as "number", "text", or "list"); the programmer develops it as they code. An abstract data type may include a constructor (such as ), which constructs one instance of the data structure, and one or more selectors (such as and ), which each select one piece of the data structure. The constructor and selector(s) together implement the abstract data type.

You can learn more about the ADT used in this project in Activity 9: Mapping over Pitches and Beats in the box labeled "How does the play song with pitches and beats block work?"

This lab can be presented without any formal teaching. Students can work directly from the activity pages with the teacher walking around asking and answering questions. If you choose to bring the class together to present or discuss some idea, consider waiting until after students have gained some hands on experience and perhaps even constructively struggled with the concept first.

Pacing

This lab is designed for 3–10 class periods (125–440 minutes).

  • Activity 1: Playing Notes: 10–20 minutes
  • Activity 2: Visualizing Loops: 10–20 minutes
  • Activity 3: Programming Styles: 15–40 minutes
  • Activity 4: Mystery Song Game: 10–50 minutes
  • Activity 5: Beat Repeat: 20–40 minutes
  • Activity 6: Storing Songs: 10–30 minutes
  • Activity 7: Song Mapper: 10–50 minutes
  • Activity 8: Songs with Pitches and Beats: 10–50 minutes
  • (Optional) Activity 9: Mapping over Pitches and Beats: 15–60 minutes

Activity 1: Playing Notes. 

  • Materials: Playing Notes student page
  • Prepare:
    • Students will need headphones or speakers for this lab. If you have both options available, consider the pros and cons for your classroom of sharing creations (speakers) vs. a quieter classroom (headphones).
  • Learning Goals:
    • Play a musical note using a command blocks.
    • Play a song by sequencing several command blocks together.
    • Gain informal experience with iteration.
  • Activity Plan:
    • Students explore a variety of sound blocks and use them to create a simple song. They are offered the opportunity to explore iteration by using the repeat block before it's formally introduced. (10-20 minutes)
  • Tips:
    • You may wish to reserve time for students to present their songs to small groups or the class.
  • Standards: None covered.
  • Solutions: The example songs on the student activity page serve as model submissions.

Activity 2: Visualizing Loops. 

  • Materials: Visualizing Loops student page
  • Prepare:
    • Before students begin working on this page, you may wish to project the Für Elise Example Song and play it with Visible Stepping turned on () to offer students a visual preview of this Snap! feature as well as of nesting loops.
  • Learning Goals:
    • Use the Snap! debugging tool Visible Stepping.
    • Use and nest loops to control the sequence of commands.
  • Activity Plan:
    • Using Visible Stepping: Students explore and discuss Snap!'s Visible Stepping tool and try it out with their own Snap! song project. (5-10 minutes)
    • Looping Your Song: Students use the repeat and forever blocks to loop one or more pieces of their song and even explore nesting one repeat block inside another like the example provided on the activity page. (5-10 minutes)
  • Tips:
    • When learning the forever block, there is always a chance that some student will struggle to stop the loop. As the text on the student activity page describes, they can either click the red stop sign () to stop all scripts or click the script that is running to stop it.
  • Standards:
    • 2-AP-12: Design and iteratively develop programs that combine control structures, including nested loops and compound conditionals.
  • Solutions: The example song on the student activity page serves as a model submission.

Activity 3: Programming Styles. 

  • Materials: Programming Styles student page
  • Prepare:
    • This is the primary page where students learn the difference between functional and imperative programming. If the distinction is still unclear to you, review the student-facing activity page before offering it to students.
  • Learning Goals:
    • Understand the difference between reporters and commands in Snap!.
    • Understand the difference between functional and imperative programming styles.
    • Create a command block.
  • Activity Plan:
    • Commands Versus Reporters: Students learn about functional programming, which uses primarily reporters, and imperative programming, which uses commands. (5-10 minutes)
    • Creating a Command Block: Mirroring their first experience creating a reporter block in Unit 1 Lab 2 Activity 2: Refining the Story, students create their first command block, mystery song, which hides the code for a song so the user can guess what it is. (10-20 minutes)
    • Optionally, students can create additional command blocks with different mystery songs. This activity can serve as differentiation either for students moving through the material more quickly or for those in need of extra practice. (5-10 minutes, if used)
  • Tips:
    • You may wish to wrap up class with either or both:
      • A brief review discussion of the three types of blocks presented on this page: reporters, predicates (a special type of reporter), and commands
      • An opportunity for several students to play their mystery song for the class to guess
  • Standards: None covered.
  • Solutions: No example solution is necessary. Students should create a mystery song block that plays a song when run.

Activity 4: Mystery Song Game. 

  • Materials: Mystery Song Game student page
  • Prepare:
    • You may wish to review the hidden text before exercise 1 on the student activity page about the difference between the reporter if then else and the command if else and about the difference between the reporter ask and the built-in pair ask and wait and answer.
  • Learning Goals:
    • Use a conditional together with a predicate to test whether the user's answer is correct.
  • Activity Plan:
    • Students read about the reporter versions of the if and ask blocks and then learn about the command versions of these blocks. They create a script to play a mystery song, accept a guess from the user, and confirm whether the user's guess is correct. (10-20 minutes)
    • Optionally, students can add additional if else blocks to offer users credit for alternative spellings. This is a good differentiation task for students moving through the material more quickly. (5-10 minutes, if used)
    • Optionally, students can take on a more challenging exercise: randomly choosing which mystery song to play. This is minimal guidance provided for this exercise, and it is suggested for a more intrepid programmer. (10-20 minutes, if used)
  • Tips:
    • You may wish to wrap up class with either or both:
      • A brief review discussion of the new blocks: if else and the pair ask and wait and answer
      • An opportunity for several students to play their mystery song for the class to guess
    • You may wish to end class with a review discussion of the new command blocks.
  • Standards: None covered.

Activity 6: Storing Songs. 

  • Materials: Storing Songs student page
  • Prepare:
    • Consider (or investigate) how well your students remember the concept of a local variable and whether additional review (behind what is included on this page) would support a deeper understanding of the distinction between local and global variables.
  • Learning Goals:
    • Create a global variable, and use it to store a list.
    • Use Snap! watchers to view or hide the contents of a global variable.
  • Activity Plan:
    • Students review the concept of a variable first introduced in Unit 1 Lab 2 (in Activity 3: Creating and Editing Blocks) with local variables, learn to use a global variable to store data that must be accessible anywhere in their program, and to show or hide the Snap! variable watcher on the Snap! stage. (10–20 minutes)
    • Optionally, students can use multiple variables to store pieces of a song and play them various times to create a more complex piece of music. (5-10 minutes, if used)
  • Tips:
    • Again, you may wish to offer time for students to share their creations in small groups or with the whole class.
  • Standards:
    • 2-AP-11: Create clearly named variables that represent different data types and perform operations on their values.
  • Solutions: The images on this activity page serve as example solutions.

Activity 9: Mapping over Pitches and Beats. 

  • Materials: Mapping over Pitches and Beats student page
  • Prepare:
    • This is the most challenging activity in the lab, and so it is optional; but it is also a natural extension of the activities so far. If your students are interested and prepared enough for tougher activity, read through this page before offering it to students so you are prepared to support them as they work through it.
  • Learning Goals:
    • Use the higher order function map to shift the pitches in a song stored as a table with both pitch and beat information about each note.
    • Use an abstract data type (the ♪ pitch beats constructor and the pitch from ♪ and beats from ♪ selectors) to create the input function for the map block.
  • Activity Plan:
    • Students first read a detailed passage describing code for mapping over a table of song data. (5-10 minutes)
    • They then reread and discuss the passage with a partner (5-10 minutes) before building their own version and using it to create a more complex song with the same musical sequence repeated a different pitches. (5-10 minutes)
    • Optionally, students who may be skeptical that this complex process is necessary are invited to investigate their own approach, and students are invited to try adjusting the beats instead of only the pitches in their song. (5-30 minutes)
  • Tips:
    • After students complete the activity page, consider doing one or more of the following:
      • Discussing how this map function works (reviewing the ideas presenting in the pre-exercise reading) to check for and improve student understanding (perhaps having several students explain it in their own words would be valuable)
      • Allowing several students share their creations
      • Invite students to co-create a song remix as a group
  • Standards:
    • 2-AP-13: Decompose problems and subproblems into parts to facilitate the design, implementation, and review of programs.
  • Solutions: The example song on the student activity page serves as a model submission.

Correlation with CSTA Standards 

  • 2-AP-11: Create clearly named variables that represent different data types and perform operations on their values. A variable is like a container with a name, in which the contents may change, but the name (identifier) does not. When planning and developing programs, students should decide when and how to declare and name new variables. Students should use naming conventions to improve program readability. Examples of operations include adding points to the score, combining user input with words to make a sentence, changing the size of a picture, or adding a name to a list of people.
  • 2-AP-12: Design and iteratively develop programs that combine control structures, including nested loops and compound conditionals. Control structures can be combined in many ways. Nested loops are loops placed within loops. Compound conditionals combine two or more conditions in a logical relationship (e.g., using AND, OR, and NOT), and nesting conditionals within one another allows the result of one conditional to lead to another. For example, when programming an interactive story, students could use a compound conditional within a loop to unlock a door only if a character has a key AND is touching the door.
  • 2-AP-13: Decompose problems and subproblems into parts to facilitate the design, implementation, and review of programs. Students should break down problems into subproblems, which can be further broken down to smaller parts. Decomposition facilitates aspects of program development by allowing students to focus on one piece at a time (e.g., getting input from the user, processing the data, and displaying the result to the user). Decomposition also enables different students to work on different parts at the same time. For example, animations can be decomposed into multiple scenes, which can be developed independently.
  • 2-AP-14: Create procedures with parameters to organize code and make it easier to reuse. Students should create procedures and/or functions that are used multiple times within a program to repeat groups of instructions. These procedures can be generalized by defining parameters that create different outputs for a wide range of inputs. For example, a procedure to draw a circle involves many instructions, but all of them can be invoked with one instruction, such as “drawCircle.” By adding a radius parameter, the user can easily draw circles of different sizes.
  • 2-AP-19: Document programs in order to make them easier to follow, test, and debug. Documentation allows creators and others to more easily use and understand a program. Students should provide documentation for end users that explains their artifacts and how they function. For example, students could provide a project overview and clear user instructions. They should also incorporate comments in their product and communicate their process using design documents, flowcharts, and presentations.
